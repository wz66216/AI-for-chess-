# Search Lab 中文化与界面优化设计

日期：2026-05-31

## 背景

Phase 2 Search Lab 已经提供独立的 `/search-lab` 页面，用于对比 Alpha-Beta 与 MCTS 搜索，并展示评估器、搜索树、局面检查器和本地运行历史。当前功能完整，但界面仍保留大量英文文案，视觉层级偏“开发工具堆叠”，新用户不容易理解每个控件与结果区的关系。

用户已确认选择方案 B：**中文化 + 信息层级重排**。本设计只覆盖 Search Lab 页面与其直接组件，不修改搜索算法、后端 API、默认搜索参数或旧分析页功能。

## 目标

1. 将 Search Lab 用户可见文案中文化，保留必要专业缩写，例如 FEN、MCTS、α-β、PST。
2. 将页面组织成更清晰的实验工作台：输入局面 → 配置搜索 → 选择评估器 → 开始搜索 → 查看结果、搜索树、局面检查器和运行历史。
3. 提升视觉层级，让主行动、选中状态、空状态、错误状态和搜索结果更容易扫读。
4. 保持实现范围可控：优先复用现有 React 组件与 Tailwind 样式，不引入新 UI 框架，不新增复杂动效。

## 非目标

- 不改变 `/api/whitebox/play` 请求/响应结构。
- 不修改 Alpha-Beta、MCTS 或评估器逻辑。
- 不把旧 `ChessGame` 白盒控件与 Search Lab 合并。
- 不新增多语言切换系统；本轮直接将 Search Lab 页面默认中文化。
- 不处理后端超时、节点预算、MCTS reward POV、tree payload 压缩等工程优化议题。

## 信息架构

Search Lab 页面保持独立路由，但内容区改成更明确的分区结构。

1. 页面头部
   - 标题：`搜索实验室`
   - 副标题：`探索不同搜索策略对局面评估的影响。`
   - 简短说明强调这是实验页，不是常规对弈页。

2. 左侧或上方配置区
   - `① 输入局面`：FEN 输入、默认局面说明、FEN 校验错误。
   - `② 配置搜索`：搜索算法、深度、着法排序、MCTS 参数。
   - `③ 选择评估器`：子力评估、位置表评估、综合启发式。
   - 主 CTA：`开始搜索`。

3. 右侧或下方结果区
   - 结果摘要卡片：最佳着法、评分、访问节点、剪枝节点、耗时等。
   - 搜索树：保留 ECharts 树，增加中文图例/提示。
   - 局面检查器：节点详情、路径、局面 FEN、说明性空状态。
   - 运行历史：用中文摘要展示最近运行，方便回看。

布局可以继续使用当前页面网格结构，但应通过卡片标题、说明文字、间距和背景层级强化阅读顺序。

## 中文文案规范

文案风格：短句、直接、偏产品化；避免冗长解释；保留必要英文缩写。

| 当前文案 | 目标文案 |
| --- | --- |
| `Search Lab` | `搜索实验室` |
| `Root FEN` | `起始局面 FEN` |
| `Alpha-Beta` | `α-β 搜索` |
| `MCTS` | `MCTS 蒙特卡洛树搜索` |
| `Depth` | `搜索深度` |
| `Move ordering` | `启用着法排序` |
| `Material` | `子力评估` |
| `Counts piece values only.` | `只统计双方棋子的基础价值。` |
| `PST` | `位置表评估` |
| `Adds piece-square positional bonuses.` | `在子力价值上加入棋子位置加成。` |
| `Heuristic` | `综合启发式` |
| `Adds mobility, pawn structure, and king safety.` | `综合考虑机动性、兵型与王安全。` |
| `Run Search` | `开始搜索` |
| `Position Inspector` | `局面检查器` |

错误与空状态示例：

- FEN 错误：`FEN 格式无效，请检查棋盘、行棋方、易位权利和回合数。`
- 搜索树空状态：`开始搜索后，这里会显示搜索树。点击节点可查看对应局面。`
- 局面检查器空状态：`请先在搜索树中选择一个节点。`
- 历史记录空状态：`暂无运行记录。完成一次搜索后会自动保存在此处。`
- 搜索进行中：`正在搜索，请稍候…`

## 组件设计

### `SearchLabPage.tsx`

- 将页面标题改为纯中文 `搜索实验室`。
- 增加一行副标题，说明页面用途。
- 保持路由和页面入口不变。

### `SearchWorkbench.tsx`

- 保持现有状态管理、请求流程和 local history 行为。
- 优化整体布局：配置区、结果区、树区、检查器区应有更明确的卡片边界。
- 搜索按钮使用更强的 CTA 样式；禁用状态仍需清晰可见。
- 搜索中、错误、无结果等状态使用中文提示。

### `PositionInputPanel.tsx`

- 标签改为 `起始局面 FEN`。
- 增加短说明：`输入任意合法 FEN，默认使用标准初始局面。`
- FEN 输入框保持等宽字体，增强可读性。
- 校验错误统一中文化。

### `SearchHyperparamsPanel.tsx`

- 算法按钮中文化，选中态更明显。
- Alpha-Beta 模式显示 `搜索深度` 与 `启用着法排序`。
- MCTS 模式显示 `模拟次数` 与 `探索系数`，说明应简短中文化。

### `EvaluatorSelector.tsx`

- 三个评估器改为中文标题和中文说明。
- 当前选中评估器使用高对比边框或背景；未选中评估器保持次级按钮/卡片样式。
- 保留 `PST` 缩写，但用 `位置表评估` 解释含义。

### `SearchResultSummary.tsx`

- 指标标题中文化，例如 `最佳着法`、`评分`、`访问节点`、`剪枝节点`、`搜索耗时`。
- 使用更清晰的数字卡片布局，让一次搜索的结果可快速扫读。
- 若无结果，显示中文空状态，不显示英文占位。

### `SearchTreeExplorer.tsx` 与 `TreeVisualizer.tsx`

- 外层控件中文化：例如 `显示剪枝节点` / `隐藏剪枝节点`。
- 增加轻量图例或说明：普通节点、最佳路径、剪枝节点分别代表什么。
- 不重写 ECharts 树渲染逻辑，只调整包装文案、卡片样式和说明。

### `PositionInspector.tsx`

- 标题改为 `局面检查器`。
- 空状态说明点击树节点后的行为。
- 节点路径、着法、评分、FEN 等标签中文化。

### `SearchRunHistory.tsx`

- 标题改为 `运行历史`。
- 每条历史显示中文摘要：算法、评估器、深度/迭代数、最佳着法、访问节点。
- 空状态中文化。

## 视觉方向

- 使用当前 Tailwind 样式体系，不新增设计依赖。
- 卡片使用统一圆角、边框、浅背景和一致间距。
- 页面主行动 `开始搜索` 应是最醒目的按钮。
- 选中态用于算法与评估器，视觉强度低于主 CTA，但明显高于普通按钮。
- 辅助说明文字使用较弱颜色和较小字号，避免压过主信息。
- 保持响应式可用：窄屏下配置区和结果区垂直堆叠，按钮允许换行。

## 数据流与行为

数据流不变：用户配置参数后点击 `开始搜索`，`SearchWorkbench` 调用 `/api/whitebox/play`，收到结果后更新摘要、搜索树、检查器和本地历史。

行为约束：

- 默认 FEN、默认搜索算法、默认深度、默认评估器保持不变。
- FEN 校验逻辑保持不变，只改中文提示。
- 搜索请求失败时保留现有错误处理流程，只改展示文案与样式。
- 本地历史存储方式保持不变。

## 测试与验证

实施完成后需要进行最小有意义验证：

1. 前端单元测试：`npm test`
2. TypeScript 检查：`npm run type-check`
3. Lint：`npm run lint`
4. 前端构建：`npm run build`
5. 浏览器手测 `/search-lab`：
   - 默认页面中文文案可见。
   - 输入非法 FEN 时显示中文错误，`开始搜索` 禁用。
   - 合法 FEN 可发起搜索。
   - 搜索完成后摘要、搜索树、局面检查器、运行历史均显示中文标签。

## 风险与缓解

- 中文文案较长可能导致按钮换行：通过 `flex-wrap`、合理宽度和短文案控制。
- 视觉样式改动可能影响测试中的文本断言：同步更新 Search Lab 相关测试，避免只改实现不改断言。
- 树图信息密度仍高：本轮只增加说明和图例，不扩大到树图算法或大规模重绘。
- 与旧分析页文案风格不完全一致：本轮仅优化 Search Lab，后续可单独统一全站文案。

## 验收标准

- Search Lab 页面核心用户可见英文文案已中文化，必要专业缩写除外。
- 页面视觉层级比当前版本更清晰，能看出输入、配置、结果、树图、检查器和历史的分区。
- 现有 Search Lab 功能行为保持不变。
- 所有相关前端测试、类型检查、lint、构建通过。
- 浏览器手测覆盖中文错误状态与一次成功搜索。
