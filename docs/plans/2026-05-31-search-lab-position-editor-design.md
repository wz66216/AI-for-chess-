# Search Lab 可视化摆子与单页分析工作台设计

## 背景

当前 `/search-lab` 已完成中文化，但主入口仍然是 FEN 文本输入。对于日常使用者，这种方式门槛过高，不适合作为主要交互。项目中已经存在基于 `react-chessboard` 的分析页棋盘实现，因此新的 Search Lab 设计应复用现有棋盘与棋子风格，把“可视化摆子 + 条件设定 + 搜索分析”整合到同一页面中，同时保留 FEN 作为高级入口，而不是主入口。

用户确认的目标是：棋子风格与现有分析页保持一致；右侧设置面板中的按钮都应真实可点击；用户完成初始条件设定后，通过“确认并开始计算”触发搜索；确认后用户仍可以继续移动棋盘上的棋子；与既有 Search Lab 搜索实验能力整合在一个页面中。用户额外明确：历史恢复与记忆功能不是本轮必需项，可以降级或后置。

## 设计目标

1. 用可视化摆子替代 FEN 作为 Search Lab 主入口。
2. 保持棋盘与棋子视觉风格和现有分析页一致，避免引入第二套棋盘语言。
3. 在同一页中清晰区分“局面设定”和“搜索分析”两个阶段，但不拆成 tab 或多页。
4. 支持“确认并开始计算”后继续移动棋子。
5. 支持可选的“移动后自动更新分析”模式。
6. 保留导入/复制 FEN 作为高级能力。
7. 提升搜索树可读性，避免节点过密导致不可用。

## 非目标

- 不新增新的棋盘依赖，优先复用现有 `react-chessboard`。
- 不重写后端搜索逻辑、接口协议或 evaluator / tree payload 结构。
- 不把历史恢复和持久化记忆作为本轮硬性范围。
- 不在本轮引入多页面导航改造。

## 总体方案

采用“**双阶段同页工作台**”结构。

页面仍然是一个 `/search-lab`，但视觉和交互上分成上下两大工作区：

- **上半区：局面设定工作台**
  - 左侧是主棋盘。
  - 右侧是棋子栏与局面条件设定面板。
  - 主操作是“确认并开始计算”。
- **下半区：搜索分析工作台**
  - 保留现有 Search Lab 的搜索方式、评估器、结果摘要、搜索树、局面检查器。
  - 它分析的是“最近一次确认提交的局面”，而不是未确认的实时编辑棋盘。

这种结构保留了现有 Search Lab 的核心成果，同时把可视化摆子前置为主入口，符合用户从“设局面”到“看分析”的自然心智流程。

## 页面结构

### 上半区：局面设定工作台

#### 左侧主棋盘

- 使用与分析页相同的 `react-chessboard` 风格。
- 棋盘格色沿用现有分析页的绿/米色配色。
- 棋盘始终显示当前 `editingPosition`。
- 支持两类交互：
  1. 摆子：选中右侧棋子，再点击棋盘格子。
  2. 确认后继续移动：直接在棋盘上进行合法走子。

#### 右侧设定面板

包含以下分区：

1. **棋子栏**
   - 白黑双方棋子按钮。
   - 不显示中文棋子名称，仅显示图形。
   - 选中某个棋子后进入“放置该棋子”模式。
   - 提供“取消选择”按钮。

2. **行棋方切换**
   - `白方走` / `黑方走` 二选一。

3. **易位权利切换**
   - `白方短易位`、`白方长易位`、`黑方短易位`、`黑方长易位` 四个独立 toggle。

4. **快捷操作**
   - `清空棋盘`
   - `标准开局`
   - `交换视角`

5. **导入 / 高级**
   - FEN 输入框
   - `应用 FEN`
   - `复制 FEN`

6. **提交分析区**
   - 主按钮：`确认并开始计算`
   - 辅助开关：`移动后自动更新分析`
   - 状态提示：当前棋盘是否已偏离已确认分析局面

### 下半区：搜索分析工作台

保留并适配现有 Search Lab 结构：

- 搜索方式 / 搜索参数
- 评估器选择
- 搜索结果摘要
- 搜索树
- 局面检查器

其中“运行历史”降级为**可选后置项**：如果实现成本低且不干扰主流程，可以保留；如果实现会显著拖慢本轮交付，则允许暂时不做或只保留简单占位。

## 状态模型

### `editingPosition`

表示用户当前正在编辑的棋盘状态。

它会被以下操作直接修改：

- 通过棋子栏点击后在棋盘放子
- 选中任意棋子后点击已有棋子的格子，移除原棋子并放入新棋子
- 切换行棋方
- 切换易位权利
- 清空棋盘 / 标准开局 / 应用 FEN
- 确认后继续在主棋盘上移动棋子

主棋盘始终渲染这份状态。

### `committedPosition`

表示最近一次用户点击“确认并开始计算”时提交给 Search Lab 的那份局面。

它的作用：

- 作为搜索请求输入
- 作为结果摘要、搜索树、检查器默认起点所对应的局面
- 用于判断当前编辑棋盘是否已经偏离已分析局面

### `selectedInspectorPosition`

用于局面检查器显示。

- 当用户点击“确认并开始计算”时，检查器先同步为 `committedPosition`。
- 当用户点击搜索树任意节点时，检查器切换到该节点对应局面。
- 主棋盘不因点击搜索树节点而被覆盖。

### 其他关键状态

- `selectedTrayPiece`：当前选中的右侧棋子
- `boardOrientation`：棋盘朝向
- `autoRecompute`：是否移动后自动更新分析
- `selectedTreeNode`：当前在树中选中的节点
- `searchConfig`：搜索方式、深度、评估器等参数
- `searchResult`：最近一次分析结果

## 关键交互规则

### 摆子规则

- 点击右侧棋子按钮后，进入放置模式。
- 点击空格子：放入该棋子。
- 点击已有棋子的格子：先移除原棋子，再放入当前选中的棋子。
- 不引入单独“删除模式”按钮。
- 用户可通过“取消选择”退出放置模式。

### 确认与计算规则

点击 `确认并开始计算` 时，一次性完成：

1. 将当前 `editingPosition` 写入 `committedPosition`
2. 触发一次搜索请求
3. 将检查器同步到这次确认的局面

### 确认后继续移动规则

- 点击确认后，主棋盘仍然可继续移动。
- 用户继续操作主棋盘时，修改的是 `editingPosition`。
- 若 `editingPosition !== committedPosition`，页面显示提示：
  - `当前棋盘已变更，以下结果仍基于上一次确认局面。`

### 自动更新分析规则（用户已选 C）

- 默认可关闭。
- 开启后，合法移动或有效局面修改会自动把当前 `editingPosition` 提交为新的 `committedPosition` 并重新触发分析。
- 为避免频繁请求，应加入轻量去抖/节流。

## 搜索分析联动

### 搜索参数位置

- 局面输入相关控件保留在上半区右侧。
- 搜索参数（搜索方式、深度、评估器等）保留在下半区顶部。

这样“局面输入”和“分析参数”边界清晰，不把右侧面板塞得过满。

### 搜索结果摘要

- 始终展示最近一次 `committedPosition` 的分析结果。
- 若当前棋盘已改动但未重新确认，则显示“结果仍基于上一次确认局面”的提示。

### 搜索树

搜索树需要从“技术全量展示”改为“可阅读展示”：

1. 默认只展示较浅层主分支。
2. 最优路径高亮。
3. 次级分支弱化。
4. 剪枝节点默认隐藏，按需显示。
5. 增大节点间距。
6. 容器支持滚动。
7. 点击某节点后，检查器切换到该节点局面。

### 局面检查器

- 点击确认后，检查器默认显示这次确认提交的局面。
- 点击搜索树节点后，检查器切换为该节点对应局面。
- 主棋盘始终保持当前编辑局面，不被树节点点击覆盖。

为了减少认知混淆，应增加一条提示文案，表达：

- 主棋盘显示当前编辑局面
- 检查器显示最近确认局面或当前选中搜索树节点局面

## 组件拆分建议

### 顶层容器

`SearchWorkbench` 继续担任总控容器，负责：

- 管理 `editingPosition` / `committedPosition`
- 触发搜索请求
- 管理自动更新分析开关
- 将状态分发给局面设定区与分析区

### 新增组件

1. `PositionEditorPanel`
   - 上半区总容器

2. `EditableChessboard`
   - 主棋盘
   - 摆子与确认后继续移动

3. `PieceTray`
   - 白黑棋子按钮
   - 当前选中态

4. `PositionSetupControls`
   - 行棋方、易位权利、快捷操作

5. `FenAdvancedPanel`
   - FEN 输入 / 应用 / 复制

6. `CommitAnalysisBar`
   - 确认按钮
   - 自动更新开关
   - 当前状态提示

### 复用并适配现有组件

- `SearchHyperparamsPanel`
- `EvaluatorSelector`
- `SearchResultSummary`
- `SearchTreeExplorer`
- `PositionInspector`

这些组件不重做，只调整输入来源和部分展示逻辑。

## 错误处理与边界行为

- 非法 FEN：显示中文错误，不覆盖当前棋盘。
- 搜索失败：继续沿用中文错误提示，不暴露英文异常主文案。
- 若当前树结果不存在可用节点元数据，检查器保留现有中文 fallback。
- 自动更新分析开启时，若连续移动过快，应通过去抖避免请求风暴。

## 测试策略

### 1. 编辑器交互测试

- 选中棋子后点击空格会放置棋子
- 选中棋子后点击已有棋子格会覆盖/移除原子
- 行棋方切换会更新局面
- 易位权利切换会更新局面
- 非法 FEN 显示中文错误

### 2. 确认与自动更新测试

- 点击确认后生成新的 `committedPosition`
- 自动更新关闭时，移动棋子不自动触发搜索
- 自动更新开启时，合法移动会自动触发搜索
- 当前棋盘变更但未确认时，显示“结果仍基于上一次确认局面”的提示

### 3. 搜索树与检查器联动测试

- 点击确认后，检查器同步到确认局面
- 点击搜索树节点后，检查器切换到对应节点局面
- 点击搜索树节点不会覆盖主棋盘
- 搜索树默认不应以过密全量方式展开

### 4. 可选测试（若保留简化历史）

- 如果本轮仍保留运行历史，则只验证其基本可读性和中文文案
- 不把复杂恢复能力作为本轮阻塞项

## 实施优先级

### P0（本轮必须完成）

- 可视化摆子主入口
- 与分析页一致的棋盘/棋子风格
- 右侧按钮真实可点击
- 确认并开始计算
- 确认后仍可继续移动
- 自动更新分析开关
- 搜索树可读性改进
- 检查器与确认/树节点联动

### P1（尽量完成）

- FEN 高级区完整体验
- 简化版运行历史继续保留

### P2（可后置）

- 历史恢复
- 记忆/持久化

## 结论

本设计将 Search Lab 从“FEN 驱动的搜索实验页”升级为“可视化设局 + 单页分析工作台”。它复用了现有棋盘基础和搜索分析能力，把用户真正常用的入口切换为摆子，同时通过 `editingPosition` / `committedPosition` 的双状态模型，保证“确认后仍可继续移动”这一需求在交互上顺手、在数据上清晰、在实现上可维护。
